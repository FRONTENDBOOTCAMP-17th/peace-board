## 1. 프로젝트 개요

Fetch API를 활용한 게시판 CRUD 웹 애플리케이션입니다. 회원가입, 로그인, 게시글 작성/목록/상세/수정 기능을 구현했습니다.

### 파일 구성

| 파일 | 역할 |
|------|------|
| `register.html` | 회원가입 (정규식 검증) |
| `login.html` | 로그인 (JWT 저장) |
| `list.html` | 게시글 목록 |
| `write.html` | 게시글 작성 (draft → publish) |
| `detail.html` | 게시글 상세보기 |
| `edit.html` | 게시글 수정 |

## 2. 잘한 점

### 2.1 뛰어난 UI 완성도
- `detail.html`과 `edit.html`에 Pretendard 폰트, box-shadow, border-radius, 색상 일관성 등 실무 수준의 CSS가 적용되어 있습니다.
- 버튼 hover 효과, focus 시 border 색상 변경 등 인터랙션이 세심합니다.

### 2.2 로그인 상태 기반 UI 제어 (write.html)
```javascript
async function init() {
  if (!token) {
    alert("로그인이 필요합니다!");
    document.getElementById("userInfo").textContent = "로그인 정보가 없습니다.";
    return;
  }
  // 인증 성공 시에만 폼을 표시
  document.getElementById("postForm").style.display = "block";
}
```
- 토큰이 없으면 폼 자체를 숨기는 방식이 좋습니다.
- 인증 만료 시 사용자에게 명확한 안내 메시지를 제공합니다.

### 2.3 404 에러 처리 (detail.html)
```javascript
if (res.status === 404) {
  document.body.innerHTML = `
    <div style="text-align:center; padding: 50px;">
      <h2>존재하지 않는 페이지입니다.</h2>
      <p>요청하신 게시글이 삭제되었거나 주소가 올바르지 않습니다.</p>
      <a href="list.html">목록으로 돌아가기</a>
    </div>`;
  return;
}
```
- 삭제된 글이나 잘못된 ID에 대해 사용자 친화적인 에러 페이지를 보여줍니다.

### 2.4 401 토큰 만료 처리 (list.html)
```javascript
if (res.status === 401) {
  localStorage.removeItem("token");
  location.href = "login.html";
}
```
- 토큰 만료 시 자동으로 localStorage를 정리하고 로그인 페이지로 이동합니다.

### 2.5 form 요소와 submit 이벤트 활용 (edit.html)
```javascript
document.getElementById("editForm").addEventListener("submit", async (e) => {
  e.preventDefault();
  // ...
});
```
- `<form>` 태그와 `submit` 이벤트, `required` 속성을 활용한 정석적인 폼 처리입니다.

### 2.6 BASE_URL 상수 사용 (write.html)
```javascript
const BASE_URL = "https://api.fullstackfamily.com/api/edu/ws-03a2de";
```
- API 주소를 상수로 분리한 점이 좋습니다. 다른 파일에도 적용하면 더 좋겠습니다.

### 2.7 Optional Chaining 활용 (detail.html)
```javascript
viewNickname.textContent = post.author?.nickname || "익명";
```
- `?.` 연산자로 데이터가 없는 경우를 안전하게 처리했습니다.

### 2.8 Git 워크플로우
- `develop` 브랜치를 기본 브랜치로 사용하고, 이슈 번호 기반 브랜치(`1-회원가입`, `3-로그인` 등)를 만드는 체계적인 전략입니다.
- PR 15개를 사용하여 적극적으로 코드 리뷰 프로세스를 활용했습니다.

## 3. 개선이 필요한 점

### 3.1 HTML 파일 앞에 마크다운 텍스트 — 심각

`register.html`과 `detail.html` 파일이 마크다운 헤딩으로 시작합니다.

```html
<!-- register.html -->
# register          ← ❌ HTML이 아닌 마크다운 텍스트

<!doctype html>
<html lang="ko">
```

```html
<!-- detail.html -->
# 게시글 상세조회    ← ❌ HTML이 아닌 마크다운 텍스트

<!doctype html>
<html lang="ko">
```

브라우저가 `<!doctype html>` 이전의 텍스트를 만나면 호환 모드(quirks mode)로 렌더링할 수 있습니다. 이 텍스트들을 삭제해야 합니다.

### 3.2 DOCTYPE 누락 (write.html)

```html
<!-- write.html — <!doctype html> 없음 -->
<html lang="ko">
  <head>
```

`<!doctype html>`이 없으면 브라우저가 표준 모드 대신 호환 모드로 동작하여 CSS 렌더링이 달라질 수 있습니다.

### 3.3 XSS 취약점 (list.html)

```javascript
item.innerHTML = `
  <div class="post-title">${post.title}</div>
  <div class="post-meta">
    작성자: ${post.authorNickname || "익명"} |
    작성일: ${new Date(post.createdAt).toLocaleString()}
  </div>
`;
```

API 응답의 `post.title`과 `post.authorNickname`이 `innerHTML`로 직접 삽입됩니다. 악의적인 스크립트가 실행될 수 있습니다.

참고로 `detail.html`에서는 `textContent`를 사용하여 안전하게 처리하고 있으므로, `list.html`도 동일하게 수정하면 됩니다.

### 3.4 글쓰기 링크 오류 (list.html)

```html
<!-- list.html 49행 -->
<a href="board.html" class="btn-write">글쓰기</a>
<!--      ^^^^^^^^^ 이 파일은 존재하지 않음 -->
```

실제 글쓰기 파일은 `write.html`이므로 링크가 깨져 있습니다.

### 3.5 삭제 기능 미구현 (detail.html)

```html
<button id="deleteBtn" class="btn btn-delete">삭제</button>
```

삭제 버튼이 HTML에 존재하지만, JavaScript에 `deleteBtn`의 이벤트 리스너가 없습니다. 버튼을 클릭해도 아무 동작이 일어나지 않습니다.

### 3.6 수정 기능 권한 체크 없음 (detail.html)

```html
<!-- 모든 글에 수정/삭제 버튼이 표시됨 -->
<button id="editBtn" class="btn btn-edit">수정</button>
<button id="deleteBtn" class="btn btn-delete">삭제</button>
```

다른 사람이 쓴 글에도 수정/삭제 버튼이 보입니다. 내 글인지 확인하는 로직이 필요합니다.

### 3.7 로그인 후 페이지 이동 없음

```javascript
// login.html
localStorage.setItem("token", data.data.token);
result.textContent = "로그인 성공\n토큰: " + data.data.token;
// 페이지 이동 없음
```

로그인 성공 후 토큰만 표시하고 목록 페이지로 이동하지 않습니다.

### 3.8 비동기 스타일 혼용 (login.html)

```javascript
// login.html — .then() 체인 방식
fetch(url, options)
  .then((res) => { ... })
  .then((data) => { ... })
  .catch((error) => { ... });

// 나머지 파일 — async/await 방식
const res = await fetch(url, options);
```

같은 프로젝트 내에서 두 가지 비동기 패턴이 혼용되어 있습니다. `async/await`로 통일하는 것이 좋습니다.

### 3.9 edit.html 수정 요청 에러 처리 미흡

```javascript
const editRes = await fetch(url, { method: "PUT", ... });
if (editRes.ok) {
  alert("수정되었습니다!");
} else {
  alert("수정 실패: 권한이 없거나 오류가 발생했습니다.");
}
// ❌ try/catch 없음 — 네트워크 에러 시 크래시
```

`fetch` 호출이 `try/catch` 없이 사용되어, 네트워크 오류 시 에러가 처리되지 않습니다.

### 3.10 console.log 잔존

```javascript
// list.html 82행
console.log("데이터 확인:", result);

// detail.html 159행
console.log("받아온 데이터:", result);
```

디버깅용 `console.log`가 남아 있습니다. 배포 전에 제거하는 것이 좋습니다.

## 5. Git 활용

### 브랜치 전략

```
main (안정 버전)
  └── develop (기본 브랜치)
        ├── 1-회원가입
        ├── 3-로그인
        ├── 4-게시글-생성하기
        ├── 5-게시글-목록보기
        └── 6-게시글-상세보기
```

- `main` / `develop` 분리와 이슈 번호 기반 브랜치 네이밍은 실무에서 사용하는 Git Flow와 유사합니다.

### PR 활용

- 총 15개의 PR을 생성하여 코드 리뷰 프로세스를 적극 활용한 점이 우수합니다.
- 다른 팀 대비 가장 체계적인 Git 활용을 보여줍니다.

### 커밋 메시지

- `feat:`, `modify:`, `docs:` 접두어를 사용하고 있으나, 일부 커밋("수정", "파일 추가")은 내용이 부족합니다.
- 예: "수정" → "fix: API 슬러그 주소 오류 수정"

## 7. 우선 수정 사항

1. **HTML 파일 앞 마크다운 텍스트 제거** — `register.html`, `detail.html`
2. **write.html에 DOCTYPE 추가** — `<!doctype html>`
3. **글쓰기 링크 수정** — `board.html` → `write.html`
4. **삭제 기능 구현** — `deleteBtn` 이벤트 리스너 추가
5. **XSS 수정** — `list.html`의 `innerHTML` → `textContent` 또는 `createElement`
6. **수정/삭제 버튼 권한 체크** — 내 글에만 버튼 표시
